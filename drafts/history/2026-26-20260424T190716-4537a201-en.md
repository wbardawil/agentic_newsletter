## THE INSIGHT

You bought the platform. You ran the pilot. The team loved the demo. Eight months later, nobody opens it. According to [HR Executive](https://hrexecutive.com/how-hr-can-break-the-tech-regret-cycle/), nearly three-quarters of US small and mid-market businesses are re-evaluating their current tech investments, with poor organizational fit named as the primary cause.

Most advisors call this a change management problem. Send people back to training. Rewrite the rollout plan. Hire a new project lead. That is the wrong diagnosis. The software did not fail because the team resisted it. It failed because the operating model behind it was never written down. You bought a system to automate a process that existed only in senior people's heads. The vendor sold you capabilities. You tried to install them on top of memory.

Here is the reframe. Technology does not create an operating model. Technology encodes the operating model you already have. If the operating model lives only in people, the software has nothing to attach to. The rollout looks like a training failure. It is actually a sequencing failure. Operating Model OS comes before Technology OS. Every time the sequence is inverted, the tech investment becomes the expensive way to discover what the operating model was supposed to be.

What I use with clients is a filter I call the Pre-Buy Readiness Test. Before any material technology purchase, three conditions must be true. First, the process the software is meant to encode exists as a written document. Not a vendor flowchart. A plain description the person running the process would sign their name to. Second, a new hire could run that process from the document without shadowing the senior person for a month. Third, the document names who decides when the process breaks and which decisions never leave the owner.

If any of the three fail, the software will fail after purchase. Not because of vendor quality. Because the check was cut before the artifact existed.

This week, do not start with the next purchase. Start with the last one. Pull the software your team stopped using. Write one page. What process was that tool supposed to encode. What document should have existed first. What part of that process still lives in one person's head. Share the page with the senior person who owns the work. Their edits will tell you where the operating model is missing. That same page is the brief for whatever you buy next. If you cannot write it, you are not buying software. You are buying the invoice for an exercise you have not started.

## THE TOOL

**The One-Page Pre-Buy Brief** is a document written before any material tech purchase, naming the process to be encoded, who owns it today, and what part of it still lives in someone's head. It matters because the tech regret cycle is a purchase-sequence problem, not a vendor problem. Build it in [Notion](https://www.notion.so) or a single Google Doc. The tool does not matter. The discipline does.
